# Logos — AI 对话沉淀与日报生成

## 核心原则

每一次有实质产出的 AI Coding 对话都是项目知识资产。Logos 的职责是将本轮对话中的需求、决策、产物、待办和经验，整理成可追溯的日报记录。

硬性约束：

- 不得编造对话中没出现的事实、文件、决策或结果。
- 不得遗漏用户明确要求记录的信息。
- 不得记录 API Key、Token、Cookie、密码、私钥、客户隐私或未经确认的商业敏感原文。
- 不覆盖已有日报；同日内容始终追加。
- 如果对话没有实质技术内容，回复“本次对话无实质技术内容，建议跳过记录”，不要写文件。

## 工作流（三步，不可跳过）

### 第一步：回顾对话

从当前上下文窗口中的完整对话历史提取：

1. **用户原始需求**：每条用户消息的核心诉求。
2. **AI 做了什么**：搜索、诊断、代码修改、文件创建、测试、设计决策。
3. **关键决策**：技术选型、架构调整、方案取舍及原因。
4. **产物清单**：新建、修改、删除了哪些文件，作用是什么。
5. **验证结果**：运行了哪些命令，结果是什么；没有验证也要明确说明。
6. **待办/遗留问题**：用户提到但未处理，或因上下文不足无法确认的事项。
7. **敏感信息**：识别需要脱敏或不应写入日报的内容。

如果上下文被压缩、截断或无法确认事实，只能记录确定信息，并在“待办 / 遗留”中写明缺口。

### 第二步：生成摘要

按 `references/template.md` 生成结构化摘要。

要求：

- 每段摘要控制在 20-50 行。
- 技术决策必须写“为什么”。
- 产物清单必须使用文件路径；没有文件产物时写“无文件变更”。
- 验证结果必须写清楚命令和结果；未运行验证时写“未运行验证”。
- 对敏感信息使用 `[REDACTED]`，不要保留原文。
- 不要把多天内容混到一个文件。

### 第三步：写入文件

写入路径：

```text
{项目根目录}/docs/daily/YYYY-MM-DD.md
```

写入规则：

- 写文件前先确认 `docs/daily/` 存在；不存在则创建。
- 当日文件不存在：创建文件头，再写入第一段摘要。
- 当日文件已存在：读取已有内容，追加到末尾。
- 段间分隔线为单独一行 `---`，前后保留空行。
- `第 N 次对话` 的 `N` 从当日文件中已有的最大编号递增；没有已有编号时使用 `1`。
- 开始/结束时间优先使用宿主工具提供的会话时间；不可得时使用当前本地时间，并在摘要中避免假装知道真实开始时间。
- 如果发现当前对话已经被记录过，不重复追加；可回复“本轮对话已记录，跳过重复写入”。

## 自动触发机制

Logos 自身不安装 hook。自动触发需要宿主 AI 工具支持 SessionEnd、Stop hook、定时任务或 automation。

### 方案 1：Hook 触发（推荐）

如果 AI 工具支持 SessionEnd / Stop hook，可在对话结束时提示执行本 Skill：

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "command": "请执行 logos skill，总结本轮有实质内容的 AI Coding 对话"
      }
    ]
  }
}
```

### 方案 2：定时调度

每日固定时间触发时，只能总结当前可访问的对话上下文。不要声称已经读取了宿主工具中不可访问的历史会话。

### 方案 3：手动触发

用户说以下任意内容时执行：

- `记录本次对话`
- `总结对话`
- `沉淀今天的内容`
- `写日报`
- `/logos`

## 不做什么

- 不编造不存在的操作、文件、测试或决策。
- 不跳过用户明确要求记录的事项。
- 不省略技术决策的原因。
- 不记录敏感信息原文。
- 不把多天内容混到同一个文件。
- 不在纯闲聊中强制记录。
- 不覆盖已有日报。

## 质量检查

写入后快速自检：

- 日报路径是否是 `docs/daily/YYYY-MM-DD.md`。
- 新建文件是否包含文件头。
- 追加内容是否用 `---` 分隔。
- 标题编号是否递增。
- 是否包含需求、决策、摘要、产物、验证、待办和经验。
- 是否去除了敏感信息。
